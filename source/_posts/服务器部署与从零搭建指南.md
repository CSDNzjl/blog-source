本文说明当前 **hongguang（虹光铸造全生命周期管理系统）** 在服务器上如何协同工作，以及若从零搭建一套类似技术栈网站时的完整流程。

> 说明：访问地址 `192.168.201.40:93` 中的 **93 端口** 属于服务器上的 **Web 服务器（通常为 Nginx）** 配置，不在本仓库源码中；下文根据现有前端打包方式与常见部署实践进行说明。

## 1. 整体架构一览

```
┌─────────────┐     :93 (Nginx 静态资源)      ┌──────────────────────────┐
│   浏览器     │ ──────────────────────────► │  dist/（Vue 打包后的 HTML/JS） │
└──────┬──────┘                               └──────────────────────────┘
       │
       │  AJAX：VUE_APP_BASE_API（打包时写入）
       │  例：http://192.168.201.40:3000/hongguang-xxx/...
       ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  Spring Cloud Gateway（hongguang-gateway）  端口 3000                      │
│  按 URL 路径前缀 /hongguang-{服务名}/ 转发到对应微服务                        │
└──────┬──────────────────────────────────────────────────────────────────┘
       │ 通过 Nacos 服务发现定位实例
       ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ auth-server  │  │ user-common  │  │ smelt        │  │ form ...     │
│ (JAR 进程)   │  │ (JAR 进程)   │  │ (JAR 进程)   │  │ (JAR 进程)   │
└──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘
       ▲                    ▲                    ▲
       │    注册 / 拉取配置 │
       └────────────────────┴──────────────────────────────────────────
                            Nacos（:8848）
                            命名空间：hongguang
                            配置分组：HONGGUANG_GROUP / COMMON_GROUP
```

**技术栈摘要**

| 层级 | 技术 |
|------|------|
| 前端 | Vue 2 + Element UI + Vue CLI 3，打包产物 `dist/` |
| 网关 | Spring Cloud Gateway，`server.port=3000` |
| 微服务 | Spring Boot 2.3 + Spring Cloud Alibaba，各业务独立 JAR |
| 注册与配置 | Nacos（服务发现 + 集中配置） |
| 数据库等 | MySQL、Redis、Elasticsearch 等（连接信息在 Nacos 配置中） |

## 2. 用户访问前端的流程（:93）

### 2.1 发生了什么

1. 用户在浏览器输入：`http://192.168.201.40:93`
2. 请求到达服务器 **93 端口** 上的 Web 服务（一般为 **Nginx**）。
3. Nginx 将站点根目录指向前端 **`dist` 目录**（打包后的 `index.html`、`static/` 等）。
4. 浏览器加载 SPA，后续路由由 Vue Router 在浏览器内完成（`history` 或 `hash` 模式需与 Nginx 的 `try_files` 配置一致）。

### 2.2 前端如何更新

在开发机 `hongguang-web` 目录：

```bash
# 安装依赖（首次）
npm install
# 或 pnpm install

# 生产打包（读取 .env.production）
npm run build:prod
```

打包完成后，将 **`dist` 目录下全部文件** 上传到服务器 Nginx 配置的 `root` 路径（覆盖旧文件即可），**无需重启 Java 服务**。

### 2.3 Nginx 配置示例（93 端口，仅供参考）

实际路径以运维环境为准，例如 `/usr/share/nginx/html/hongguang`：

```nginx
server {
    listen 93;
    server_name 192.168.201.40;

    root /usr/share/nginx/html/hongguang;   # 此处放 dist 内容
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;   # SPA 路由回退
    }

    # 可选：静态资源缓存
    location /static/ {
        expires 7d;
    }
}
```

修改 Nginx 配置后执行：`nginx -t && nginx -s reload`。

## 3. 前端如何知道后端地址（核心）

**结论：前端不会在运行时查 Nacos；网关地址在「打包时」写进前端代码。**

### 3.1 环境变量（打包时注入）

生产环境配置在 `hongguang-web/.env.production`：

```properties
ENV = 'production'
VUE_APP_BASE_API = 'http://192.168.201.40:3000'
```

执行 `npm run build:prod` 时，Vue CLI 会把 `VUE_APP_BASE_API` **编译进** JS  bundle。

### 3.2 Axios 统一入口

`hongguang-web/src/utils/request.js` 创建 axios 实例：

```javascript
baseURL: process.env.VUE_APP_BASE_API
```

因此所有接口请求的「主机 + 端口」都是 **`http://192.168.201.40:3000`**（网关），而不是各个微服务的真实端口。

### 3.3 业务 API 只写「服务路径前缀」

`hongguang-web/src/utils/define.js` 定义各微服务在网关上的路径前缀，例如：

| 常量 | 路径前缀 | 对应后端服务名 |
|------|----------|----------------|
| `authService` | `/hongguang-auth-server` | hongguang-auth-server |
| `userService` | `/hongguang-user-common` | hongguang-user-common |
| `formService` | `/hongguang-form` | hongguang-form |
| `smeltService` | `/hongguang-smelt` | hongguang-smelt |
| … | `/hongguang-{模块}` | 与 `spring.application.name` 一致 |

业务 API 示例（`hongguang-web/src/api/user.js`）：

```javascript
// 实际请求 URL =
// http://192.168.201.40:3000 + /hongguang-auth-server/oauth/token
url: define.authService + '/oauth/token'
```

### 3.4 一次完整请求链路（以登录为例）

```
浏览器
  → POST http://192.168.201.40:3000/hongguang-auth-server/oauth/token
网关 hongguang-gateway (:3000)
  → 根据路径 /hongguang-auth-server/** 匹配路由
  → 从 Nacos 发现 hongguang-auth-server 实例
  → 转发到某台机器上的 auth JAR 进程
auth-server
  → 校验用户，返回 token
浏览器
  → 后续请求 Header: Authorization: bearer {token}
```

### 3.5 开发环境 vs 生产环境

| 环境 | 前端运行方式 | API 地址来源 |
|------|--------------|--------------|
| 本地开发 `npm run dev` | Vue devServer（默认 3000 需注意端口冲突） | `.env.development` 中的 `VUE_APP_BASE_API`；也可用 `vue.config.js` 的 `/dev` 代理 |
| 生产部署 | Nginx 提供静态 `dist` | `.env.production` 打包进代码 |

**重要：** 若服务器 IP 或网关端口变更，必须 **改 `.env.production` 后重新 `build` 并上传 dist**，仅改服务器上的旧 dist 无效。

### 3.6 例外：部分功能直连其他端口

少数页面在代码中写死了其他服务地址（未走统一网关），例如：

- 积木报表：`http://192.168.201.40:83/jmreport/...`
- BI 大屏：`http://192.168.201.40:8370/...`

这些需单独部署对应服务并在防火墙放行端口。

## 4. 后端与 Nacos 如何协同

### 4.1 Nacos 的两个作用

1. **服务注册与发现**：各微服务启动后注册到 Nacos，网关按服务名负载均衡转发。
2. **集中配置**：数据库、Redis、网关路由等放在 Nacos 配置中心，各服务 `bootstrap.yaml` 拉取。

典型配置（以网关为例，`hongguang-gateway/src/main/resources/bootstrap.yaml`）：

```yaml
spring:
  application:
    name: hongguang-gateway
  cloud:
    nacos:
      config:
        server-addr: kocel-fom-cloud-zayton-register:8848
        namespace: hongguang
        group: HONGGUANG_GROUP
        prefix: ${spring.application.name}
server:
  port: 3000
```

各业务服务类似，并常引用公共配置：

- `nacos-registry-common.yaml`（注册、网关路由等）
- `druid-common.yaml`（数据源）
- `logback-common.yaml`（日志）

`server-addr` 中的主机名 `kocel-fom-cloud-zayton-register` 一般在服务器 `/etc/hosts` 或 Docker 网络中解析到 **192.168.201.40:8848**（以实际运维为准）。

### 4.2 网关路由规则（概念）

Spring Cloud Gateway 通常配置为：

- **Path 断言**：`/hongguang-auth-server/**` → 服务 `hongguang-auth-server`
- **Path 断言**：`/hongguang-smelt/**` → 服务 `hongguang-smelt`

路由定义多在 **Nacos 中的 `hongguang-gateway.yaml` 或 `nacos-registry-common.yaml`**，不在本仓库单独维护一份静态路由表。  
前端路径前缀必须与 **`spring.application.name`** 一致，否则网关无法匹配。

### 4.3 微服务之间的调用

服务间通过 **服务名** 调用（经 Nacos 解析），例如 OAuth 用户信息：

```yaml
security.oauth2.resource.user-info-uri: http://kocel-fom-cloud-zayton-gateway:3000/hongguang-auth-server/user
```

即：内部也经网关访问认证服务，而不是写死 IP。

## 5. 后端 JAR 在服务器上的部署方式

项目提供统一部署脚本目录：`hongguang-server/deploy/hongguang/`。

### 5.1 目标目录

```
/home/ifom/hongguang/          # 部署根目录（INSTALL.txt 约定）
├── gateway/
│   ├── hongguang-gateway-1.0.0-SNAPSHOT.jar
│   ├── app.env
│   ├── start.sh / stop.sh / restart.sh
│   └── logs/
├── auth-server/
├── form/
├── smelt/
│   └── ...
├── service-order.conf         # 启动顺序
├── start-all.sh
├── stop-all.sh
└── bin/common.sh
```

### 5.2 打包与放置 JAR

在开发机各模块目录执行 Maven 打包，例如：

```bash
cd hongguang-server/hongguang-gateway
mvn clean package -DskipTests
```

将 `target/*.jar` 复制到服务器对应子目录，文件名与 `app.env` 中 `JAR_NAME` 一致。

`gateway/app.env` 示例：

```bash
JAR_NAME=hongguang-gateway-1.0.0-SNAPSHOT.jar
JAVA_OPTS="-Xms512m -Xmx1024m"
SPRING_PROFILES=dev    # 或 prod，决定加载哪套 Nacos 配置
```

### 5.3 启动顺序

`service-order.conf` 规定顺序（节选）：

1. `gateway`（网关最先，对外 3000）
2. `auth-server`（认证）
3. `api`、`form`、`smelt` 等业务服务
4. …

```bash
cd /home/ifom/hongguang
chmod +x start-all.sh stop-all.sh
./start-all.sh          # 按顺序启动全部
./status-all.sh         # 查看 PID
./gateway/restart.sh    # 单服务重启
```

启动命令实质为：

```bash
java -Xms512m -Xmx1024m -jar hongguang-gateway-1.0.0-SNAPSHOT.jar --spring.profiles.active=dev
```

### 5.4 前置依赖

在启动业务 JAR 之前，需保证已运行：

| 组件 | 典型端口 | 说明 |
|------|----------|------|
| Nacos | 8848 | 配置 + 注册中心 |
| MySQL | 3306 | 业务库（连接串在 Nacos） |
| Redis | 6379 | 缓存/会话（视配置） |
| Elasticsearch | 9200 | 日志/检索（部分模块） |

| 端口 | 角色 | 说明 |
|------|------|------|
| **93** | 前端静态站点 | Nginx 托管 `dist`，用户访问入口 |

## 7. 从零搭建类似系统的完整流程

以下假设你要新建一套「Vue 前端 + Spring Cloud 微服务 + Nacos + 网关」的管理系统。

### 阶段一：基础设施

1. **准备服务器**（Linux，建议 CentOS 7+ / Ubuntu 20+）。
2. **安装 JDK 8**（本项目基于 Java 8）。
3. **安装 MySQL 8**，创建业务库与用户。
4. **安装 Redis**（若使用缓存/分布式锁）。
5. **部署 Nacos**（单机或集群），创建：
   - 命名空间：如 `myproject`
   - 配置分组：`XXX_GROUP`
   - 导入或编写：`nacos-registry-common.yaml`、`druid-common.yaml` 等。
6. **（可选）Elasticsearch、XXL-Job、MinIO** 等按业务需要安装。

### 阶段二：后端微服务

1. **搭建父工程**：Maven 多模块，`spring-boot-starter-parent` + `spring-cloud-dependencies` + `spring-cloud-alibaba-dependencies`。
2. **创建模块**：
   - `xxx-gateway`：Gateway，端口 3000（或自定义）。
   - `xxx-auth-server`：OAuth2 / JWT 认证。
   - `xxx-user`、`xxx-business-...`：按领域拆分。
   - `xxx-common`：公共工具、常量。
3. **每个服务配置 `bootstrap.yaml`**：
   - `spring.application.name` = 对外路径前缀（如 `xxx-user`）。
   - `spring.cloud.nacos.discovery` / `config` 指向 Nacos。
4. **在 Nacos 配置网关路由**：`Path=/xxx-user/**` → `lb://xxx-user`。
5. **本地验证**：先起 Nacos → 网关 → 认证 → 一个业务服务，用 Postman 访问  
   `http://localhost:3000/xxx-auth-server/oauth/token`。
6. **编写 deploy 脚本**（可参考本仓库 `deploy/hongguang`）：`app.env` + `start.sh` + `service-order.conf`。
7. **服务器部署**：上传 JAR，`./start-all.sh`，确认 Nacos 控制台中服务均为 **健康**。

### 阶段三：前端

1. **初始化 Vue 2/3 项目**（本仓库为 Vue CLI 3 + Vue 2）。
2. **配置环境变量**：
   - `.env.development`：`VUE_APP_BASE_API=http://localhost:3000`
   - `.env.production`：`VUE_APP_BASE_API=http://你的服务器IP:网关端口`
3. **封装 `request.js`**：`baseURL: process.env.VUE_APP_BASE_API`。
4. **定义 `define.js`**：为每个后端服务写路径前缀 `/xxx-{服务名}`，与 `spring.application.name` **严格一致**。
5. **API 层只写相对路径**：`url: define.userService + '/user/list'`。
6. **本地开发**：`npm run dev`，确认能登录、调通接口。
7. **生产打包**：`npm run build:prod`，得到 `dist/`。
8. **Nginx 部署**：配置 `listen` 端口与 `root` 指向 `dist`，`try_files` 支持 SPA。

### 阶段四：联调与上线检查清单

- [ ] Nacos 中所有服务实例在线
- [ ] 网关 3000 可访问，`/actuator/health` 或 Swagger（若开启）正常
- [ ] 前端 dist 中打包的 `VUE_APP_BASE_API` 与现网网关地址一致
- [ ] 浏览器 F12 → Network：API 请求为 `网关:端口/服务名/...`，无 CORS 错误
- [ ] 登录拿 token 后，带 `Authorization: bearer ...` 的业务接口正常
- [ ] 防火墙放行：93（前端）、3000（网关）、8848（Nacos 管理，可限内网）
- [ ] 修改后端仅重启对应 JAR；修改前端必须重新 build 并覆盖 dist

### 阶段五：日常运维

| 操作 | 做法 |
|------|------|
| 更新前端页面 | 本地 build → 上传 dist 覆盖 |
| 更新某个微服务 | 替换对应目录 JAR → `./xxx/restart.sh` |
| 修改数据库/Redis 地址 | 在 Nacos 改配置 → 重启相关服务或开启 refresh |
| 修改网关对外端口 | 改 Nacos 或 bootstrap + 重启 gateway + **改前端 .env 并重新打包** |

## 8. 常见问题

### Q1：改了 `.env.production` 但浏览器仍请求旧地址？

必须重新执行 `npm run build:prod` 并上传新 `dist`。环境变量在构建时写入，不是运行时读取。

### Q2：前端 93 能打开，但接口 404 / 502？

1. 检查网关 3000 是否启动：`./gateway/start.sh` 或 `status-all.sh`。  
2. 检查 Nacos 中目标服务是否注册。  
3. 检查请求路径前缀是否与 `spring.application.name` 一致。  
4. 查看 `gateway/logs/console.log` 与各服务日志。

### Q3：CORS 跨域错误？

生产环境一般 **前后端不同端口**（93 vs 3000），网关需配置 CORS（本仓库 `GlobalGatewayCorsConfig` 等）。若仍报错，检查网关是否启动、是否走了网关而非直连微服务端口。

### Q4：`kocel-fom-cloud-zayton-register` 解析失败？

在服务器配置 hosts 或 Docker 网络别名，使其指向 Nacos 实际 IP。各服务 `bootstrap.yaml` 中的 `server-addr` 需与现网一致。

### Q5：Windows 上传脚本后 `set: pipefail: invalid option`？

部署脚本为 Linux bash。上传后按 `INSTALL.txt` 执行 `sed` 去除 CRLF，并 `chmod +x`。

## 10. 总结

| 问题 | 答案 |
|------|------|
| `:93` 是什么？ | 对外的 **前端静态站点**（Nginx + dist） |
| 前端如何知道后端地址？ | **打包时** 通过 `VUE_APP_BASE_API` 指向 **网关 :3000**，业务 URL 再加 `/hongguang-服务名/...` |
| Nacos 管什么？ | 微服务 **注册发现** + **集中配置**；前端 **不连接** Nacos |
| 后端如何部署？ | 各模块打成 JAR，放到 `/home/ifom/hongguang/{服务名}/`，脚本启停 |
| 请求如何到达具体服务？ | 浏览器 → 网关按路径转发 → Nacos 查实例 → 对应 JAR |

按本文流程，可以从零复现一套与当前虹光项目同架构的 Web 系统；具体 IP、目录、Nginx 路径以你们服务器实际运维配置为准，替换文档中的 `192.168.201.40` 即可。
