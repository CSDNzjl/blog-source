Hexo 博客源码备份与 GitHub Pages 部署指南

本文整理自 Hexo + GitHub Pages 搭建个人博客 类教程的常见流程，并补充源码备份说明。很多教程只教 hexo deploy，容易让人误以为 GitHub 上已有完整项目——实际上往往只有编译后的静态网站。

1. 先搞清楚：GitHub 上到底有什么

Hexo 博客在本地的典型目录结构：

执行 hexo deploy 时，Hexo 会把 public/ 里的内容推送到 username.github.io 仓库，供 GitHub Pages 展示。

内容 本地 username.github.io 仓库
source/_posts/*.md 文章原文 ✅ ❌
_config.yml 配置 ✅ ❌
package.json、主题源码 ✅ ❌
编译后的 index.html、CSS、JS ✅（在 public/） ✅

结论： hexo deploy 部署的是成品网站，不是写作源码。若只把 deploy 仓库当作备份，换电脑后无法直接继续用 Markdown 写博客。

2. 推荐方案：两个仓库分工

仓库 用途 更新方式
CSDNzjl.github.io 存放编译后的静态网站，供 GitHub Pages 访问 hexo deploy
blog-source（名称自定） 存放 Hexo 源码，便于换机、协作、回滚 git push

两个仓库互不替代：一个给人看，一个给自己改。

5. 日常写作与发布流程

建议养成习惯：先 push 源码，再 deploy 网站。这样 GitHub 上始终有最新文章原文。

6. 换电脑后如何恢复

若 hexo deploy 报错，检查：

  • _config.yml 中 deploy.repository 是否指向正确的 username.github.io
  • 本机是否已配置 GitHub SSH 密钥（ssh -T git@github.com 测试）

7. 常见问题

Q1：hexo deploy 时 Git 只显示少量文件变更，是不是丢内容了？

不是。Git 只提交有变化的文件，仓库里仍是完整网站。这是正常现象。

Q2：GitHub 上为什么看不到 .md 文件？

因为 deploy 仓库里只有 HTML。要看 Markdown 原文，请打开源码仓库（如 blog-source）。

Q3：能否只用一个仓库？

可以，但不推荐新手使用。常见做法是：

  • main 分支：放源码
  • gh-pages 分支：放编译结果

需要额外改 deploy 配置，且与「用户名.github.io」默认用 main 展示页面的习惯容易混淆。两个仓库更清晰。

Q4：文章里的图片怎么管理？

开启 post_asset_folder: true 后，每篇文章可有独立资源目录。图片应放在 source/ 下，随源码仓库一起 git push；hexo generate 会一并打包进 public/ 并由 deploy 推上 Pages。

8. 检查清单

换电脑前确认：

  • 源码仓库已 push 最新 source/_posts/ 文章
  • _config.yml、主题自定义已提交
  • 文章引用的本地图片已在 source/ 中并提交
  • 知道 username.github.io 与 blog-source 各自用途

虹光项目：服务器部署架构与从零搭建指南

本文说明当前 hongguang（虹光铸造全生命周期管理系统） 在服务器上如何协同工作，以及若从零搭建一套类似技术栈网站时的完整流程。

说明：访问地址 192.168.201.40:93 中的 93 端口 属于服务器上的 Web 服务器（通常为 Nginx） 配置，不在本仓库源码中；下文根据现有前端打包方式与常见部署实践进行说明。

1. 整体架构一览

技术栈摘要

层级 技术
前端 Vue 2 + Element UI + Vue CLI 3，打包产物 dist/
网关 Spring Cloud Gateway，server.port=3000
微服务 Spring Boot 2.3 + Spring Cloud Alibaba，各业务独立 JAR
注册与配置 Nacos（服务发现 + 集中配置）
数据库等 MySQL、Redis、Elasticsearch 等（连接信息在 Nacos 配置中）

2. 用户访问前端的流程（:93）

2.1 发生了什么

  1. 用户在浏览器输入：http://192.168.201.40:93
  2. 请求到达服务器 93 端口 上的 Web 服务（一般为 Nginx）。
  3. Nginx 将站点根目录指向前端 dist 目录（打包后的 index.html、static/ 等）。
  4. 浏览器加载 SPA，后续路由由 Vue Router 在浏览器内完成（history 或 hash 模式需与 Nginx 的 try_files 配置一致）。

4.1 Nacos 的两个作用

  1. 服务注册与发现：各微服务启动后注册到 Nacos，网关按服务名负载均衡转发。
  2. 集中配置：数据库、Redis、网关路由等放在 Nacos 配置中心，各服务 bootstrap.yaml 拉取。

前端路径前缀必须与 spring.application.name 一致，否则网关无法匹配。

7. 从零搭建类似系统的完整流程

以下假设你要新建一套「Vue 前端 + Spring Cloud 微服务 + Nacos + 网关」的管理系统。

阶段一：基础设施

  1. 准备服务器（Linux，建议 CentOS 7+ / Ubuntu 20+）。
  2. 安装 JDK 8（本项目基于 Java 8）。
  3. 安装 MySQL 8，创建业务库与用户。
  4. 安装 Redis（若使用缓存/分布式锁）。
  5. 部署 Nacos（单机或集群），创建：
    • 命名空间：如 myproject
    • 配置分组：XXX_GROUP
    • 导入或编写：nacos-registry-common.yaml、druid-common.yaml 等。
  6. （可选）Elasticsearch、XXL-Job、MinIO 等按业务需要安装。

阶段二：后端微服务

  1. 搭建父工程：Maven 多模块，spring-boot-starter-parent + spring-cloud-dependencies + spring-cloud-alibaba-dependencies。
  2. 创建模块：
    • xxx-gateway：Gateway，端口 3000（或自定义）。
    • xxx-auth-server：OAuth2 / JWT 认证。
    • xxx-user、xxx-business-...：按领域拆分。
    • xxx-common：公共工具、常量。
  3. 每个服务配置 bootstrap.yaml：
    • spring.application.name = 对外路径前缀（如 xxx-user）。
    • spring.cloud.nacos.discovery / config 指向 Nacos。
  4. 在 Nacos 配置网关路由：Path=/xxx-user/** → lb://xxx-user。
  5. 本地验证：先起 Nacos → 网关 → 认证 → 一个业务服务，用 Postman 访问
    http://localhost:3000/xxx-auth-server/oauth/token。
  6. 编写 deploy 脚本（可参考本仓库 deploy/hongguang）：app.env + start.sh + service-order.conf。
  7. 服务器部署：上传 JAR，./start-all.sh，确认 Nacos 控制台中服务均为 健康。

阶段三：前端

  1. 初始化 Vue 2/3 项目（本仓库为 Vue CLI 3 + Vue 2）。
  2. 配置环境变量：
    • .env.development：VUE_APP_BASE_API=http://localhost:3000
    • .env.production：VUE_APP_BASE_API=http://你的服务器IP:网关端口
  3. 封装 request.js：baseURL: process.env.VUE_APP_BASE_API。
  4. 定义 define.js：为每个后端服务写路径前缀 /xxx-{服务名}，与 spring.application.name 严格一致。
  5. API 层只写相对路径：url: define.userService + '/user/list'。
  6. 本地开发：npm run dev，确认能登录、调通接口。
  7. 生产打包：npm run build:prod，得到 dist/。
  8. Nginx 部署：配置 listen 端口与 root 指向 dist，try_files 支持 SPA。

阶段四：联调与上线检查清单

  • Nacos 中所有服务实例在线
  • 网关 3000 可访问，/actuator/health 或 Swagger（若开启）正常
  • 前端 dist 中打包的 VUE_APP_BASE_API 与现网网关地址一致
  • 浏览器 F12 → Network：API 请求为 网关:端口/服务名/...，无 CORS 错误
  • 登录拿 token 后，带 Authorization: bearer ... 的业务接口正常
  • 防火墙放行：93（前端）、3000（网关）、8848（Nacos 管理，可限内网）
  • 修改后端仅重启对应 JAR；修改前端必须重新 build 并覆盖 dist

阶段五：日常运维

操作 做法
更新前端页面 本地 build → 上传 dist 覆盖
更新某个微服务 替换对应目录 JAR → ./xxx/restart.sh
修改数据库/Redis 地址 在 Nacos 改配置 → 重启相关服务或开启 refresh
修改网关对外端口 改 Nacos 或 bootstrap + 重启 gateway + 改前端 .env 并重新打包

8. 常见问题

Q1：改了 .env.production 但浏览器仍请求旧地址？

必须重新执行 npm run build:prod 并上传新 dist。环境变量在构建时写入，不是运行时读取。

Q2：前端 93 能打开，但接口 404 / 502？

  1. 检查网关 3000 是否启动：./gateway/start.sh 或 status-all.sh。
  2. 检查 Nacos 中目标服务是否注册。
  3. 检查请求路径前缀是否与 spring.application.name 一致。
  4. 查看 gateway/logs/console.log 与各服务日志。

Q3：CORS 跨域错误？

生产环境一般 前后端不同端口（93 vs 3000），网关需配置 CORS（本仓库 GlobalGatewayCorsConfig 等）。若仍报错，检查网关是否启动、是否走了网关而非直连微服务端口。

Q4：kocel-fom-cloud-zayton-register 解析失败？

在服务器配置 hosts 或 Docker 网络别名，使其指向 Nacos 实际 IP。各服务 bootstrap.yaml 中的 server-addr 需与现网一致。

Q5：Windows 上传脚本后 set: pipefail: invalid option？

部署脚本为 Linux bash。上传后按 INSTALL.txt 执行 sed 去除 CRLF，并 chmod +x。

10. 总结

问题 答案
:93 是什么？ 对外的 前端静态站点（Nginx + dist）
前端如何知道后端地址？ 打包时 通过 VUE_APP_BASE_API 指向 网关 :3000，业务 URL 再加 /hongguang-服务名/...
Nacos 管什么？ 微服务 注册发现 + 集中配置；前端 不连接 Nacos
后端如何部署？ 各模块打成 JAR，放到 /home/ifom/hongguang/{服务名}/，脚本启停
请求如何到达具体服务？ 浏览器 → 网关按路径转发 → Nacos 查实例 → 对应 JAR

按本文流程，可以从零复现一套与当前虹光项目同架构的 Web 系统；具体 IP、目录、Nginx 路径以你们服务器实际运维配置为准，替换文档中的 192.168.201.40 即可。
